```txt
# 核心风格：专业术语 + 口语化叙述

这是你的核心翻译准则：你必须在“技术精准”和“语气自然”间找到完美平衡。

1.  **技术术语 (Tech Terms)**：
    在谈论技术概念、架构、代码和术语时，必须 100% 专业、精准、符合行业标准。
2.  **非技术措辞 (Wording & Tone)**：
    在叙述、连接和解释时，必须使用**口语化、非正式、轻松**的风格。使用自然的日常对话方式，就像在和同事进行技术评审 (Code Review) 或在 Slack 里聊天。

**风格示例**：

* **中文原文**：“为了优化性能，该函数被设计为异步处理高并发请求。因此，调用者不应立即假定操作已完成。”

* **[禁止] 过度书面语**：`In order to optimize performance, this function is designed to process high-concurrency requests asynchronously. Therefore, the caller should not immediately assume the operation is complete.`

* **[标准] 混合风格 (你应采用的)**：`So, to boost performance, this function handles high-concurrency requests asynchronously. This just means the caller shouldn't assume the operation is done right away.`

# 工作流程

## 1. 评估语境（最重要）

你必须先判断中文内容的**预期使用场景**。

* **[行动]**：如果语境不明确（例如：只有一个词“部署”），**必须先询问**：“这个词/这句话的预期使用场景是什么？（例如：UI 按钮、文档描述，还是回复用户提问？）”
* **[例外]**：如果我已提供语境（例如：`回复用户: 提交`），则直接进入第2步。

## 2. 翻译原则 (按语境调整)

* **(A) 回复用户 / 博客 / 讨论 (默认场景)**：
    * **原则**：严格遵循上述的 **[核心风格]**（专业术语 + 口语化叙述）。
* **(B) 技术文档/API参考**:
    * **原则**：同样遵循 **[核心风格]**，但可以稍微正式一点（例如，使用完整的句子），但依然要避免僵硬的学术腔。
* **(C) UI/界面字符串** (如：按钮, 标签):
    * **原则**：此时**忽略**口语化。规则是：极度简洁。多用动词原形或名词。
    * *例：中文“删除文件” -> (按钮) `Delete File`*
* **(D) Git Commit 信息**:
    * **原则**：此时**忽略**口语化。遵循 [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) 规范。
    * *例：中文“修复了删除文件的bug” -> `fix: Correctly handle file deletion errors`*
* **(E) 代码注释**:
    * **原则：** 简洁明了，帮助理解即可。
    * *例：中文“删除文件” -> (注释) `// Handle file deletion`*
```

#### 新增一、 新用户引导优化：通过注册问卷精准画像

**目标：** 在不显著影响注册转化率的前提下，于用户注册流程中置入简短问卷，以建立初步的用户画像，为后续的个性化引导和商业转化奠定基础。

##### 核心逻辑

* **时机选择：** 在用户通过邮箱授权、注册成功后，跳转至应用初始页前插入问卷。此阶段用户对产品抱有最高期待，愿意用少量信息换取使用权，是数据收集的“黄金窗口”。
* **数据驱动决策：** 严格监控问卷加入前后从“注册成功”到“进入应用”的转化率。设定一个可接受的流失率阈值（例如，相较于无问卷流程，转化率下降不超过10%），以此来动态调整问卷的问题数量和形式。

##### 建议补充的问卷问题

1.  **哪一种符合你的情况**
    * *说明*：了解用户的身份构成。
    * *选项*：个人 / 独立开发者、企业雇员、政府部门雇员、高校组织（课题、学术研究）

1.  **您的身份角色？ (Role)**
    * *说明*：了解用户的职能，直接关联其使用场景。
    * *选项*：开发者/工程师、产品经理、运营/市场、业务/销售、学生/研究员、个人爱好者、其他

2.  **您的团队规模？ (Team Size)**
    * *说明*：评估潜在商业价值（ARR/LTV）的关键指标，也可以作为 pricing 团队人数的参考。
    * *选项*：小于 3 人、3 - 5 人、6 - 10 人、10 + 人

3.  **您计划使用 Dify 构建什么？ (Primary Use Case)**
    * *说明*：这是实现场景化引导的核心，应设为多选题。
    * *选项（示例）*：
        * 构建智能客服与对话机器人 (Intelligent Customer Service)
        * 内容创作与自动化营销 (Content Creation & Marketing Automation)
        * 企业内部知识库与AI助理 (Internal Knowledge Base & AI Assistant)
        * 数据分析与报告生成 (Data Analysis & Reporting)
        * 我还在探索... (Still Exploring...)

##### 后续动作补充

* **个性化邮件跟进 (Onboarding Email Sequence)**：注册一周后，可根据用户在问卷中的画像，自动触发个性化邮件。例如，对“未进行任何操作”的用户发送引导邮件，附上对应场景的教程链接和个人客服联系方式，主动破冰。

#### 二、 场景化运营：从“模板市场”到“预设工作流”

##### 核心痛点

当前的应用模板市场解决了“插件如何组装”的问题，但没有解决用户更核心的“在哪里用”的问题。用户从察觉需求到找到并安装正确模板的路径过长，每一步都存在流失。

##### 解决方案

* **变“人找模板”为“场景找人”**：根据上一步问卷中用户选择的场景，当用户首次进入应用时，**不再是一个空空如也的画布**，而是为其**默认预置1-2个最相关的场景化工作流 (Workflow)**。
* **统一内部认知：** 定义核心主推场景，需要产品、市场、销售、运营等所有部门达成共识。各部门围绕核心场景协同工作：
    * **产品**：持续迭代场景化工作流，使其更易用、更强大。
    * **市场/运营**：围绕场景创作内容（博客、教程、案例），进行精准推广。
    * **销售**：聚焦于场景的商业价值，向潜在客户展示解决方案。

##### 建议补充

* **用户生成场景 (UGC)**：发起运营活动，鼓励用户分享自己创造的工作流如何解决特定场景问题。获选的优秀案例不仅能丰富场景库，更能成为极具说服力的“社会认同”证据。

#### 三、 社区建设：构建信息沉淀与流量新入口

### 核心问题
当前以 IM（即时通讯工具）为主的交流方式，信息碎片化、难检索、难复用，且对未来的 LLM（大语言模型）流量入口不友好。

##### 解决方案

* **建立官方论坛 (Forum)**：采用 Discourse 或 Flarum 等现代开源论坛框架。

##### 论坛带来的核心价值

1.  **知识沉淀与SEO**：将重复提问和优质解答结构化，形成可被搜索引擎收录的帖子，极大提升网站的 SEO 权重和自然流量。
2.  **LLM友好型数据**：结构化的问答和文章是未来训练垂直领域LLM或被其检索引用的最佳数据源。
3.  **用户数据金矿**：在论坛注册时，可鼓励用户填写公司、职位等信息进行“企业认证”。这既能收集到高价值的销售线索，又能形成“XX 公司也在用 Dify”的真实背书。
4.  **活动与互动中心**：所有运营活动都有了最佳的落地页面，便于用户参与和讨论。

##### 数据指标优化建议：从数据收集到洞察行动


###### 1. 用户画像与细分 (User Persona & Segmentation)

* **目标**：不仅知道“谁”来了，更要知道“谁是我们的理想客户(ICP - Ideal Customer Profile)”。
* **收集数据点**：公司信息、团队规模、用户角色。
* **其他关键指标**
    * **理想客户画像匹配度 (ICP Match Rate)**：
        * **衡量什么**：注册用户中有多少比例符合我们定义的高价值客户画像？（例如：“大于 10 人的企业团队中的开发者”）。
        * **为什么重要**：这个指标可以评估市场推广渠道的质量。如果匹配度低，说明吸引来的用户与产品目标用户错位，付费转化会很困难。
    * **用户来源渠道分析 (Channel Analysis)**：
        * **衡量什么**：不同渠道（论坛、SEO、广告、技术博客）来的用户，其 ICP 匹配度和后续的“激活率”分别是多少？
        * **为什么重要**：帮助运营和市场团队将预算和精力集中在能带来高质量用户的渠道上。

###### 2. 用户激活与采纳 (User Activation & Adoption)

* **目标**：衡量用户是否体验到了产品的“啊哈时刻”(Aha! Moment)，即真正理解了产品的核心价值。
* **收集数据点**：使用状态（是否上线）、业务周期（试用到上线时长）。
* **其他关键指标**
    * **激活率 (Activation Rate)**：
        * **衡量什么：** 新用户在注册后 7 天内，完成了至少一次“关键行为”的比例。对于 Dify，这个“关键行为”可以定义为：**“创建了第一个应用”**或**“成功进行了一次 API 调用”**。
        * **为什么重要**：这是衡量产品新用户引导是否成功的最核心指标。激活率低说明产品上手门槛高或价值不清晰。
    * **价值实现时间 (Time to Value - TTV)**：
        * **衡量什么：** 用户从注册到完成“关键行为”的平均时长。（这正是您“试用到上线时长”的量化版）。
        * **为什么重要**：TTV 越短，用户越快能感受到产品价值，留存和付费的可能性就越大。
    * **产品采纳率 (Product Adoption Rate)**：
        * **衡量什么**：活跃用户中，使用了至少3个核心功能（例如：Agent 构建、RAG 管道、数据集管理）的比例。
        * **为什么重要**：衡量用户对产品理解的深度。用户使用的功能越多，说明产品对他的价值越大，迁移成本越高，越不容易流失。

###### 3. 用户留存与互动 (User Retention & Engagement)

* **目标**：衡量用户是否愿意长期使用我们的产品，以及使用的深度如何。
* **您已有的数据点**：Cloud 使用情况判断、论坛互动。
* **建议补充的关键指标**：
    * **次周/次月留存率 (Week 2 / Month 2 Retention Rate):**
        * **衡量什么**：通过“同期群分析(Cohort Analysis)”，查看某一周注册的新用户，在接下来的一周/一月后，还有多少比例的人在继续使用。
        * **为什么重要**：这是衡量产品长期吸引力和健康度的“生命线”。
    * **用户粘性 (Stickiness - DAU/MAU Ratio)**：
        * **衡量什么**：日活跃用户数(DAU) / 月活跃用户数(MAU)。这个比率越高，说明用户使用产品的频率越高。
        * **为什么重要**：高粘性是产品成为用户日常工作流一部分的标志。
    * **Dify 核心互动指标 (Dify Core Engagement Metrics)**：
        * **衡量什么**：比如“月均 API 调用次数”、“创建的应用数量”、“处理的消息量”等。
        * **为什么重要**：这些是比“登录次数”更有价值的互动指标，直接反映了用户对产品核心价值的消耗。

##### 总结表格

| 阶段 (Stage) | 核心问题 (Key Question) | 关键指标 (Key Metrics) |
| :--- | :--- | :--- |
| **用户画像** | 来的用户对吗？ | ICP 匹配度、渠道质量 |
| **用户激活** | 用户体验到价值了吗？ | 激活率、产品采纳率、TTV |
| **用户留存** | 用户还回来吗？ | 次周/月留存率、DAU/MAU 比率、核心功能使用量 |
| **商业价值** | 产品能赚钱吗？ | 免费到付费转化率、客户生命周期价值(LTV) |

#### 场景化运营

对应上文的场景 1、2、3…，这里主要是解决未来（可能现在已经遇到）的应用分发问题。随着生态越来越丰富，用户找到其合适的插件（扩展）会变得越来越困难。

用户的路径为：

```
需求察觉 - 初步需求方案（这里假定为 插件 n） - 寻找应用市场（marketplace）对应插件 - 安装
```

每一步操作，都是流失。用户首先不一定察觉到他的需求，其次不一定知道 marketplace 有对应的解决方案。所以 **预设分发** 是个对用户来说，操作最少；对我们来说，流失（因为操作带来的流失）最低的方案。

场景化运营，不同于现在的模版。模版解决的是扩展如何组装的问题，但仍然没解决一个问题：“**用在哪里**”的问题。产品、销售、市场和运营，就未来期望用户使用的插件，搭建其场景，即：我们主推何种场景，一定要所有部门都知晓这件事，才能更好地统一朝一个方向用力。
* 销售负责场景的盈利；
* 市场和运营负责场景的推广；
* 产品负责合理化场景，满足用户所需；

> 场景可以后面慢慢丰富，对应的上面问卷 quiz 的第四题应该是多选题。

一旦用户勾选了所需场景之后，不用选择对应模版（现在用户注册完成，初始页面空空如也）。直接默认预置场景 1、2、3 等等的 workflow。

这里注意数据场景埋点。

此外，除了产品经历设定场景之外，也可以开展运营活动，让用户参加，发布他/她用的 workflow，解决何种场景，并附上效果图。非常简单的 3 步骤，完成了用户场景背书——这里转到下一个话题，用户活动场所。

#### 用户活动地、数据聚合地 —— 论坛

目前，用户同用户、用户同官方的交流渠道以 IM（discord、微信、telegram）为主。这里有一个问题。信息沉淀，以及信息复用，以及 llm 友好。

未来，可能是 10 年，可能是 1、2 年，流量的入口一定是 chatbot 口中，通过 llm 给用户传递出来信息。既然如此，我们如何面向 llm 获得流量，就是尽量将结构化的数据喂给模型。

##### 论坛选择

不多说，国外主流：https://www.discourse.org/ 、 https://github.com/flarum/framework （这两个都是开源的项目）

##### 论坛的好处

* 解决重复提问；
* 结构化问题、文章可以喂给 llm，供其学习；
* 问题、文章，都是 seo 收录的内容。论坛的帖子多了，会提高整个站点的 seo 排名；
* 活动有对应的落地页，更方便的用户互动；
* **用户数据收集**：可以让用户注册是，填写相关的公司信息之类的完成认证。至于要收集何种数据，权看运营操作所需；

> 企业认证的一个好处是，用户在论坛互动时，实时的所在公司也在为 dify 背书——佐证某家企业在用，以及的确有不少企业在用。

#### 数据

这里的数据专指企业数据、政府数据等，这里主要讲如何收集。收集完，如何分析看市场和运营所需。

* 公司信息
    * 收集方式：论坛注册收集、用户 1v1 收集（cloud 注册不收集公司信息，因为过于敏感会劝退用户）

    * 收集方式：cloud、论坛注册

    * 收集方式：论坛注册、cloud 使用情况判断、1v1 确认

##### 页面加载（不利于 SEO）

官网页面加载非常卡顿，SEO 的有一项指标就是页面加载速度。前端实现我不懂，所以我丢给 gemini pro 分析了。TL;DR：过于炫技，**设计美感大于用户体验**。

下面是原文：

```
您好。我分析了 https://dify.ai/zh 这个网站。

首先，您“非常卡顿，加载异常慢”的感觉是非常真实的，并且有充分的技术原因可以解释。

总的来说，这个网站在设计上追求了非常炫酷和现代的视觉效果（例如复杂的动画和视频背景），但在前端性能优化上做得不够理想，导致了“设计感”和“流畅度”之间的脱节。

下面是我分析出的几个导致卡顿和缓慢的主要原因：

1. 大量且复杂的动画效果 (导致“卡顿”)
这是您在滚动页面时感觉到“卡顿”和“掉帧”的最主要原因。

实现方式： 该网站使用了大量的滚动触发式动画（即您滚动到某个位置，元素才开始移动、淡入或变形）。这些动画虽然在视觉上很吸引人，但如果优化不当，会极大地消耗您电脑的CPU和GPU资源。

技术细节： 流畅的网页动画应该只改变 transform (位移、缩放、旋转) 和 opacity (透明度) 这两个属性，因为它们通常能由GPU加速，不会引起页面“重排”。而 Dify 网站上的一些动画可能改变了元素的 width, height, margin 等属性，这会强制浏览器进行大量的重新计算和布局，从而导致卡顿。

2. 大体积的视频和图片资源 (导致“加载异常慢”)
这是您首次打开网站时感觉“慢”的主要原因。

背景视频： 网站首页（以及其他页面）使用了大面积的背景视频或高分辨率的动态图形。这些视频文件通常是 MB 级别，是拖慢初次加载速度的最主要“元凶”。

图片优化不足： 网站上的一些图片可能没有经过充分的压缩，或者没有使用更现代、更高效的图片格式（如 WebP），导致不必要的文件体积过大。

3. 庞大的JavaScript脚本 (导致“可交互时间”变长)
该网站使用了 Next.js (一个 React 框架) 构建，这本身是一个非常现代的选择。但是：

JS包体积： 现代框架需要加载JavaScript文件来让页面变得可交互。如果代码没有经过良好的分割和优化，首次加载的JS文件会非常大。这些JS需要被下载、解析和执行，这个过程会阻塞页面的响应，让您感觉“点什么都没反应”。

客户端渲染： 很多内容的显示依赖于JS的执行，这也增加了用户能真正开始与页面交互前的等待时间。

4. 过多的第三方脚本
为了实现网站分析、在线客服等功能，网站还加载了多个第三方服务脚本。例如：

Google Analytics (网站分析)

Intercom (右下角的在线客服)

Hotjar (用户行为热力图分析)

每一个脚本都意味着一次额外的网络请求和一段需要执行的JavaScript代码，它们会互相竞争网络和CPU资源，进一步拖慢网站的整体速度。

结论：它的前端实现和设计合理吗？
这是一个需要从两个角度来看的问题：

从“品牌展示”和“吸引眼球”的角度看，这种设计是‘可以理解的’。
Dify.ai 作为一个技术前沿的AI公司，希望通过这种高度动态化、充满科技感的设计来展示其技术实力和现代感，给访问者留下深刻的第一印象。

但是，从“用户体验”和“性能最佳实践”的角度看，这种实现方式是‘不合理的’，并且有巨大的优化空间。
对于一个官网来说，清晰、快速地传递核心信息（产品是什么、能做什么）比炫技更重要。过度的卡顿和缓慢的加载会劝退一部分耐心不足的潜在用户，甚至可能因为糟糕的“核心网页指标”而对SEO产生负面影响。

总结： Dify.ai 的网站在视觉设计上投入了很多，但在实现时为了这些效果牺牲了大量的性能，导致了您感觉到的卡顿和缓慢。这是一个典型的**“重设计、轻优化”的案例。它在美学上或许‘合理’，但在技术实现和用户体验上不够‘合理’**。
```
